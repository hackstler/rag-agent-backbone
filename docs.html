rag-agent-backbone

arquitectura completa · capas de sistema · flujo de datos

Capa 0
Entrada & Configuración

Antes de que llegue cualquier petición, el servidor tiene que arrancar y saber cómo comportarse. Esta capa hace dos cosas: levantar el servidor HTTP y cargar todos los parámetros que controlan el comportamiento del agente. Si algo falla aquí, el proceso muere antes de abrir el puerto — mejor fallar rápido que fallar en mitad de una llamada.

src/index.ts
Punto de entrada del servidor

Es el primer fichero que se ejecuta cuando arranca la aplicación. Hace tres comprobaciones antes de abrir el puerto: verifica que existe la clave de Google (sin ella no hay embeddings ni LLM), activa la extensión pgvector en PostgreSQL (sin ella no hay búsqueda vectorial), y finalmente abre el servidor en el puerto 3000.

También registra tres middlewares globales que se aplican a todas las peticiones sin excepción: un logger que imprime cada request en consola, cabeceras de seguridad HTTP estándar, y la política CORS que controla qué dominios pueden llamar a la API.

Si cualquier error no controlado llega hasta aquí, lo captura, lo loguea y devuelve un 500 genérico sin exponer información interna.

src/config/rag.config.ts
Configuración central del agente

Un único objeto — ragConfig — que contiene todos los parámetros que controlan cómo funciona el sistema. El resto del código lee de aquí en lugar de tener números o strings hardcodeados.

topK (por defecto 5): cuántos fragmentos de documentos recuperar por cada pregunta. Más = más contexto pero más coste.
similarityThreshold (0.7): puntuación mínima para considerar un fragmento relevante. Un fragmento con 0.5 de similitud se descarta directamente.
queryEnhancement: si la búsqueda inicial falla, qué estrategia usar para reformular la pregunta y buscar de nuevo.
enableReranking: si después de recuperar fragmentos se aplica un segundo modelo más potente para reordenarlos.

Los useCasePresets son configuraciones predefinidas por tipo de caso de uso. Si se configura el agente como "customer-support", automáticamente activa multi-query y reranking sin tener que cambiar nada más.

src/config/tools.config.ts
Registro de herramientas activas

Controla qué herramientas tiene disponibles el agente. Si una herramienta no está en este fichero marcada como enabled: true, el agente no sabe que existe y nunca intentará llamarla.

searchDocuments siempre está activa — es la herramienta principal de búsqueda en los documentos propios.
searchWeb solo se activa si existe la variable de entorno PERPLEXITY_API_KEY. Si no hay clave, la herramienta no se registra y el sistema no tiene fallback a internet.

Esta separación permite habilitar o deshabilitar funcionalidades solo cambiando variables de entorno, sin tocar código.

Arranque en local
Cómo levantar el sistema

Se necesitan dos terminales y la base de datos corriendo:

Terminal 1 — API: npm run dev en la raíz del proyecto. Levanta el servidor con hot reload en localhost:3000.

Terminal 2 — Frontend: npm run dev dentro de la carpeta frontend/. Levanta la interfaz visual en localhost:5173.

Base de datos: podman start ecstatic-meninsky_postgres_1 si el contenedor ya existe, o docker-compose up para crearlo por primera vez.

Capa 1
API HTTP

La API es la puerta de entrada al sistema. Todo lo que quiera hablar con el agente — una app móvil, una web, un script de automatización — lo hace a través de estos endpoints HTTP. Están construidos con Hono, un framework web ligero optimizado para edge computing. Cada endpoint valida la entrada con Zod antes de procesarla: si el formato es incorrecto, devuelve un error 400 inmediatamente sin llegar nunca al agente.

POST /chat — src/api/chat.ts
Pregunta al agente y espera la respuesta completa

El endpoint principal. Recibe la pregunta del usuario, la pasa al agente, y devuelve la respuesta completa cuando el agente termina de pensar.

Flujo interno: valida el body con Zod → busca o crea una conversación en base de datos → llama a ragAgent.generate() que puede tardar varios segundos → extrae las fuentes de los pasos intermedios del agente → guarda la pregunta y la respuesta en base de datos → devuelve todo junto.

La respuesta incluye el texto de la respuesta, las fuentes con sus fragmentos, el ID de conversación para continuar el hilo, y metadatos como el modelo usado y cuántos fragmentos se recuperaron.

GET /chat/stream — src/api/chat.ts
Pregunta al agente con respuesta token a token

Mismo proceso que POST /chat pero usando SSE (Server-Sent Events). En lugar de esperar la respuesta completa, el servidor va enviando eventos en tiempo real mientras el agente trabaja.

El orden de los eventos es siempre el mismo: primero llega un evento sources con los fragmentos encontrados (antes de que el LLM empiece a escribir), luego llegan múltiples eventos text con cada fragmento de texto a medida que el modelo lo genera, y al final llega done.

Si algo falla durante el stream, se emite un evento error seguido igualmente de done, para que el cliente siempre sepa cuándo ha terminado.

sources → text → done Content-Type: text/event-stream
POST /ingest — src/api/ingest.ts
Añadir documentos a la base de conocimiento

Recibe la ruta de un fichero local o una URL y lo procesa para indexarlo. Acepta PDFs, Markdown, HTML, texto plano, ficheros de código, o cualquier URL pública.

El proceso es asíncrono en el sentido de que puede tardar bastante si el documento es grande (hay que dividirlo, generar embeddings para cada fragmento y guardarlos). Cuando devuelve respuesta, el documento ya está indexado y disponible para búsquedas.

Acepta un parámetro orgId para asociar el documento a un cliente o proyecto concreto, de modo que las búsquedas se puedan filtrar por organización.

GET /health — src/api/health.ts
Verificar que el sistema funciona

Hace una query mínima a la base de datos (SELECT 1) para verificar que la conexión funciona. Si la DB está caída, devuelve 503. Si todo está bien, devuelve 200 con un timestamp.

Los sistemas de monitorización y los balanceadores de carga llaman a este endpoint periódicamente para saber si el servidor está sano. En Railway o cualquier plataforma cloud, si este endpoint falla repetidamente, la plataforma reinicia el contenedor automáticamente.

Validación con Zod: todos los endpoints validan el input antes de procesarlo. Las queries de chat tienen un máximo de 10.000 caracteres. Los IDs de conversación deben ser UUIDs válidos. Si llega un body malformado o un parámetro con el tipo incorrecto, el servidor responde 400 sin llegar a tocar el agente ni la base de datos.
Capa 2
Agente Mastra

El agente es el cerebro del sistema. No es un flujo rígido de pasos — es un LLM (Gemini) que lee la pregunta del usuario, decide qué herramientas llamar y en qué orden, interpreta los resultados, y genera la respuesta. Mastra es el framework que conecta el LLM con las herramientas, gestiona la memoria de conversación, y proporciona observabilidad sobre lo que hace el agente en cada paso.

src/agent/rag-agent.ts
Definición del agente

Aquí se ensambla todo: el LLM, las herramientas disponibles, las instrucciones de comportamiento, y la memoria. El resultado es ragAgent, un objeto que expone dos métodos: generate() para respuestas completas y stream() para respuestas en tiempo real.

Las instrucciones son el system prompt: un conjunto de reglas numeradas en orden de prioridad estricta. Primero responde saludos sin llamar tools. Segundo, para preguntas reales siempre llama searchDocuments. Tercero, si encuentra fragmentos los usa y no llama searchWeb. Solo si no hay fragmentos usa la búsqueda web. Nunca inventa información. Siempre cita las fuentes.

El orden importa: sin estas reglas el LLM toma atajos, como responder de su conocimiento general en lugar de buscar en los documentos.

src/agent/rag-agent.ts — Memory
Memoria de conversación

El agente recuerda los mensajes anteriores de cada conversación. La memoria está respaldada en PostgreSQL (en un schema separado llamado "mastra" para no mezclarse con las tablas de la aplicación).

Configurada con ventana fija: guarda los últimos N mensajes (N = windowSize × 2 porque cada turno tiene un mensaje de usuario y uno de asistente). Los mensajes más antiguos se descartan sin resumir.

Esto permite que el agente tenga contexto de la conversación: si el usuario pregunta "¿y qué más?" o "explícame el punto 2", el agente sabe a qué se refiere porque tiene los mensajes anteriores disponibles.

src/agent/tools/search-documents.ts
Herramienta principal: buscar en documentos

La herramienta que el agente llama para cualquier pregunta factual. El LLM decide cuándo llamarla y con qué parámetros — puede reformular la pregunta antes de buscar si cree que una versión diferente dará mejores resultados.

Internamente delega en runRetrievalPipeline(), que hace el trabajo pesado: convertir la pregunta en vector, buscar en pgvector, expandir si hace falta, reranquear. La tool solo formatea los resultados para devolvérselos al LLM.

Acepta filtros opcionales: orgId para buscar solo en documentos de un cliente concreto, documentIds para restringir la búsqueda a documentos específicos.

src/agent/tools/search-web.ts
Herramienta de fallback: búsqueda web

Solo entra en juego cuando searchDocuments devuelve cero fragmentos relevantes. Llama a la API de Perplexity (modelo Sonar) que sintetiza resultados de múltiples fuentes web en una respuesta coherente con sus citas.

Si no hay PERPLEXITY_API_KEY configurada, la herramienta devuelve available: false y el agente lo interpreta como "búsqueda web no disponible".

El agente siempre informa al usuario cuando la respuesta viene de internet en lugar de los documentos propios.

¿Cómo decide el agente qué tool llamar? El LLM recibe el system prompt con las reglas, la pregunta del usuario, y la lista de tools disponibles con sus descripciones. Elige qué llamar basándose en las instrucciones y en el contenido de la pregunta. Después de recibir el resultado de la tool, decide si ya tiene suficiente información para responder o si necesita llamar a otra tool. Todo este proceso ocurre dentro de un único ragAgent.generate().
Capa 3a
Pipeline de Retrieval

RAG significa Retrieval-Augmented Generation: generar respuestas aumentadas con información recuperada. Esta capa es la parte de "recuperación". Su trabajo es, dada una pregunta en texto natural, encontrar los fragmentos de documentos más relevantes para esa pregunta. No busca por palabras clave — busca por significado semántico. Y si la búsqueda inicial no da suficientes resultados, tiene estrategias para ampliar el recall automáticamente.

Paso 1
Embed
La pregunta en texto se convierte en un vector de 768 números. Este vector captura el significado semántico: preguntas sobre el mismo tema tendrán vectores parecidos aunque usen palabras distintas.
Paso 2
Retrieve
Se buscan en pgvector los fragmentos cuyo vector sea más cercano al de la pregunta. La "cercanía" se mide con distancia coseno. Solo se devuelven fragmentos con score superior al umbral configurado.
Paso 3 (condicional)
Expand
Si se recuperan menos de 3 fragmentos, se reformula la pregunta de varias maneras (multi-query, HyDE, step-back) y se repite la búsqueda. Los resultados se fusionan eliminando duplicados.
Paso 4
Rerank
Si el reranking está activado, un segundo modelo evalúa cada fragmento directamente contra la pregunta y los reordena por relevancia real. Mejora la precisión ~20-35%.
src/rag/embeddings.ts — createEmbedding()
¿Qué es un embedding y para qué sirve?

Un embedding es una representación matemática del significado de un texto. Se convierte el texto en una lista de 768 números donde textos similares producen listas de números similares.

Por ejemplo, "¿qué puedo cenar hoy?" y "ideas para la cena de esta noche" producirán vectores muy parecidos aunque no compartan ninguna palabra. Esto permite encontrar documentos relevantes aunque el usuario use un vocabulario distinto al del documento.

Usa Gemini gemini-embedding-001 por defecto. Si no hay clave de Google pero sí de OpenAI, cambia automáticamente a text-embedding-3-small. La dimensión del vector (768) tiene que coincidir con la columna en la base de datos.

src/rag/retriever.ts — retrieve()
Búsqueda vectorial en PostgreSQL

Ejecuta una query SQL especial que usa el operador <=> de pgvector para calcular la distancia coseno entre el vector de la pregunta y los vectores de todos los fragmentos almacenados.

La fórmula es 1 - distancia = similitud. Un score de 1.0 significa que los vectores son idénticos. Un score de 0.7 (el umbral por defecto) significa que son bastante similares. Por debajo del umbral, el fragmento se descarta aunque aparezca en los resultados del índice.

Solo busca en fragmentos de documentos con status = "indexed". Los documentos en proceso de ingesta o que fallaron no aparecen en las búsquedas.

src/rag/query-transformer.ts — transformQuery()
Estrategias de expansión de query

Multi-query: pide al LLM que reformule la pregunta de N maneras distintas. "¿qué recetas hay con pollo?" genera también "platos con pollo", "cómo cocinar pollo", etc. Se busca con todas y se fusionan los resultados. Mejora el recall ~20-30%.

HyDE (Hypothetical Document Embeddings): en lugar de buscar con la pregunta, genera una respuesta hipotética de ~100 palabras y la usa como query. La lógica: el embedding de una respuesta se parece más al embedding del documento que contiene la respuesta que el embedding de la pregunta.

Step-back: generaliza la pregunta a una más amplia. "¿cuántas calorías tiene una tortilla de 3 huevos?" → "información nutricional de los huevos". Recupera contexto general que puede contener la respuesta específica.

src/rag/reranker.ts — rerank()
Reordenación por relevancia real

El retrieval vectorial recupera fragmentos semánticamente similares, pero la similitud semántica no es lo mismo que "responde la pregunta". El reranker usa un modelo cross-encoder que evalúa cada par (pregunta, fragmento) de forma directa.

Cohere API: modelo rerank-english-v3.0. Calidad alta. Requiere COHERE_API_KEY. Añade ~200ms de latencia pero mejora la precisión notablemente.

Reranker local: sin API externa. Combina el score de embedding original (70%) con el solapamiento de palabras clave entre la pregunta y el fragmento (30%). Más rápido pero menos preciso.

Para que el reranker tenga buenos candidatos, el retriever recupera topK × 3 fragmentos cuando el reranking está activo, y el reranker devuelve solo los mejores topK.

src/rag/chunker.ts — chunk()
División de documentos en fragmentos

Los documentos se dividen en fragmentos antes de indexarlos porque los modelos de embedding tienen un límite de tokens, y porque fragmentos pequeños dan resultados de búsqueda más precisos que documentos completos.

Fixed (por defecto): fragmentos de tamaño fijo con overlap. El overlap (~50 tokens) garantiza que una idea que cruza el borde entre dos fragmentos aparece en ambos, evitando perder información en las juntas.

Semantic: divide por párrafos y encabezados markdown. Respeta la estructura del documento en lugar de cortar cada N caracteres.

Hierarchical: detecta secciones por encabezados, divide dentro de cada sección con fixed, y guarda el nombre de la sección en los metadatos de cada fragmento.

src/rag/interfaces.ts + adapters.ts
Contratos e implementaciones intercambiables

Los interfaces (IEmbedder, IRetriever, IReranker) definen qué métodos tiene que tener cada componente, sin decir cómo están implementados. El pipeline solo habla con estas interfaces.

Los adapters son las implementaciones concretas que cumplen esos contratos: defaultEmbedder usa Gemini, pgvectorRetriever usa PostgreSQL, defaultReranker usa Cohere o local.

Para cambiar de proveedor de embeddings solo hay que modificar adapters.ts — el pipeline, las tools, y el workflow no necesitan tocarse porque solo conocen los interfaces.

Capa 3b
Ingesta de Documentos

Para que el agente pueda responder preguntas sobre documentos, primero hay que indexar esos documentos. La ingesta es el proceso de tomar un fichero o URL, extraer su texto, dividirlo en fragmentos manejables, generar un vector para cada fragmento, y guardar todo en la base de datos. Una vez indexado, el retriever puede encontrarlo en milisegundos.

src/ingestion/loader.ts — loadDocument()
Extracción de texto según el tipo de fichero

Detecta automáticamente qué tipo de fuente es y usa el método de extracción adecuado para cada una.

PDF: usa pdfjs-dist para extraer el texto página por página. La importación es dinámica (solo se carga la librería si hay un PDF que procesar).

Markdown: lee el fichero como texto y extrae el título del primer encabezado #.

HTML/URL: usa Cheerio (el jQuery del servidor) para parsear el HTML. Elimina automáticamente elementos de navegación, scripts, estilos, cabeceras y pies de página para quedarse solo con el contenido relevante.

Código: cualquier extensión no reconocida se trata como código fuente. Añade el nombre del fichero como comentario al principio para que el agente sepa de qué fichero viene cada fragmento.

src/ingestion/processor.ts — processDocument()
El pipeline completo de ingesta

Recibe el documento cargado y lo procesa en cuatro pasos secuenciales.

1. Registro en DB: antes de hacer nada, crea un registro en la tabla documents con status = "processing". Si el proceso falla a mitad, queda registrado en base de datos con el error en los metadatos. Sin esto, no habría forma de saber qué documentos fallaron.

2. Chunking: divide el texto usando la estrategia configurada en ragConfig.

3. Embedding en lotes: genera vectores para todos los fragmentos en lotes de 20. Los lotes limitan las llamadas paralelas a la API de Gemini para no superar los rate limits. Dentro de cada lote se usa Promise.all() para paralelizar y ir rápido.

4. Insert y marcado: inserta todos los fragmentos con sus vectores en la tabla document_chunks y actualiza el documento a status = "indexed".

¿Por qué el ciclo de estados? Un documento pasa por pending → processing → indexed (o failed). El retriever solo busca en documentos con status = "indexed". Esto garantiza que un documento que se está procesando o que falló a mitad nunca aparece en los resultados de búsqueda con datos incompletos.
Capa 4
Persistencia — PostgreSQL + pgvector

Todo el estado del sistema vive en una única base de datos PostgreSQL con la extensión pgvector instalada. PostgreSQL gestiona los datos relacionales (conversaciones, mensajes, documentos). pgvector añade el tipo de dato vector y los operadores de búsqueda por similitud. Un solo DATABASE_URL lo conecta todo.

tabla: document_chunks ★
La tabla más importante del sistema

Cada fila es un fragmento de texto con su vector de embedding. La columna embedding vector(768) almacena los 768 números que representan semánticamente ese fragmento.

El índice ivfflat sobre esa columna es lo que hace la búsqueda vectorial rápida. Sin él, cada búsqueda haría un scan completo de todos los fragmentos. Con él, encuentra los más similares en milisegundos incluso con millones de filas.

El operador <=> calcula la distancia coseno entre dos vectores. Un resultado de 0 significa vectores idénticos; cuanto más cercano a 0, más similares.

tabla: documents
Registro de cada documento ingestado

Guarda la ficha de cada documento: título, fuente original (ruta o URL), tipo de contenido, y el estado actual del proceso de ingesta.

El campo orgId permite asociar documentos a un cliente o proyecto. Las búsquedas pueden filtrarse por este campo para que cada cliente solo vea sus propios documentos.

El campo chunkCount guarda cuántos fragmentos se generaron, útil para diagnóstico. El campo indexedAt registra cuándo se completó la indexación.

tablas: conversations + messages
Historial de conversaciones

Cada conversación es un hilo de mensajes. Cuando un usuario hace varias preguntas relacionadas, todas van a la misma conversación y el agente tiene contexto de las anteriores.

El campo updatedAt en conversations se actualiza en cada mensaje nuevo, lo que permite ordenar las conversaciones por actividad reciente.

Los mensajes del asistente guardan en metadata los IDs de los fragmentos que se usaron para generar la respuesta. Esto permite auditar de dónde vino cada respuesta.

La relación es en cascada: borrar una conversación borra automáticamente todos sus mensajes.

tabla: users (opcional)
Usuarios — opcional

La tabla de usuarios existe para cuando se necesita autenticación, pero no es requerida para que el sistema funcione. La mayoría de las peticiones son anónimas: el agente responde sin necesidad de saber quién pregunta.

Si se activa autenticación, los usuarios se asocian a conversaciones y se pueden filtrar documentos por usuario además de por orgId.

schema: "mastra"
Memoria del agente (Mastra)

Mastra guarda su propio historial de conversación en la misma base de datos pero en un schema separado llamado "mastra". Esto evita conflictos entre las tablas de la aplicación y las tablas internas de Mastra.

La diferencia con la tabla messages: la tabla de la aplicación guarda los mensajes para mostrarlos en el frontend. El schema de Mastra guarda el historial que el agente recibe como contexto al procesar cada pregunta.

src/db/schema.ts + client.ts
ORM Drizzle — cómo se define el schema

Las tablas se definen en TypeScript con Drizzle ORM. Drizzle genera el SQL necesario para crear y migrar las tablas.

Todas las PKs son UUIDs generados automáticamente (no enteros incrementales). Todos los timestamps incluyen timezone. Los campos JSON usan jsonb con tipos TypeScript para type-safety.

Las migraciones se aplican con npm run migrate y nunca se modifican una vez creadas — cada cambio al schema genera una nueva migración.

Capa 5
APIs Externas

El sistema delega tres tipos de trabajo en APIs externas: la generación de texto (el LLM), la generación de vectores (embeddings), y la búsqueda web de fallback. Sin estas APIs no hay inteligencia — son la capa que convierte texto en comprensión semántica y genera las respuestas. Cada API requiere su propia clave de acceso en el fichero .env.

Google Gemini API — GOOGLE_API_KEY
LLM y embeddings (la API más importante)

Se usa para dos propósitos distintos del mismo proveedor:

Generación de texto (gemini-2.5-flash): el LLM que razona, decide qué tools llamar, y genera la respuesta final. Es el modelo más capaz de la familia Flash de Gemini — rápido, eficiente y con ventana de contexto grande.

Embeddings (gemini-embedding-001): convierte texto en vectores de 768 dimensiones. Se usa tanto al ingestar documentos (para indexar cada fragmento) como al buscar (para vectorizar la pregunta). El parámetro taskType le indica a la API si el vector es para indexar (RETRIEVAL_DOCUMENT) o para buscar (RETRIEVAL_QUERY) — Gemini optimiza el vector según el uso.

GOOGLE_API_KEY requerida sin esta key el servidor no arranca
Cohere API — COHERE_API_KEY (opcional)
Reranking de calidad con cross-encoder

El modelo rerank-english-v3.0 de Cohere es un cross-encoder: en lugar de comparar vectores, evalúa cada par (pregunta completa, fragmento completo) de forma directa y asigna un score de relevancia.

Esto es mucho más preciso que la similitud vectorial porque el modelo entiende la relación semántica entre la pregunta y el contenido del fragmento, no solo si "hablan de lo mismo".

Si no hay COHERE_API_KEY, el sistema usa automáticamente el reranker local que combina el score de embedding con solapamiento de palabras clave. Funciona, pero con menor precisión.

Perplexity API — PERPLEXITY_API_KEY (opcional)
Búsqueda web como fallback

Solo entra en juego cuando la base de conocimiento propia no tiene información relevante (el retriever devuelve 0 fragmentos). En ese caso, el agente llama a Perplexity para buscar en internet.

Perplexity usa su modelo Sonar, que sintetiza resultados de múltiples fuentes web en una respuesta coherente con las citas de origen incluidas.

Si no hay PERPLEXITY_API_KEY, la herramienta searchWeb no se registra y el agente responde que no tiene información disponible cuando el retrieval falla.

solo si searchDocuments devuelve 0 chunks
Variables de entorno — .env
Resumen de claves necesarias

Mínimo para funcionar en local:

DATABASE_URL — conexión a PostgreSQL con pgvector
GOOGLE_API_KEY — para Gemini (LLM + embeddings)

Opcionales para funcionalidad extra:

COHERE_API_KEY — reranking de calidad (si no, usa local)
PERPLEXITY_API_KEY — búsqueda web fallback
GEMINI_MODEL — cambiar el modelo de Gemini (por defecto: gemini-2.5-flash)
ALLOWED_ORIGINS — dominios permitidos en CORS (producción)
